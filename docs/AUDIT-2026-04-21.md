## 1. Resumo executivo

O sistema está **operacional e respondendo publicamente** — todos os 12 serviços principais estão UP (Traefik, Nginx sidecar, PostgreSQL 16 + pgvector 0.8.1, Redis 7, Evolution API v2.3.4, dois MCPs, WireGuard, API Node.js, Agent Python, vsftpd), restart counts estão em zero para a quase totalidade dos containers Docker, as rotas HTTPS retornam 200 nos hostnames `agente.forumtelecom.com.br` e `agenteevo.forumtelecom.com.br`, e ambos os certificados Let's Encrypt R12 estão válidos até 2026-05-22 (31 dias). Apesar disso, **6 bugs críticos** foram encontrados com potencial de comprometer confidencialidade, integridade ou continuidade: o daemon `cron` do sistema não está instalado (entries em `/etc/cron.d/` nunca executam), as portas 4000 (API Node) e 8000 (Agent Python) estão abertas publicamente via UFW — bypassando o proxy reverso — há 6 vulnerabilidades HIGH em dependências Node (incluindo `path-to-regexp` com DoS e `tar` com path traversal), o middleware `requireAdmin` está definido no código mas não é aplicado em endpoints que executam comandos de sistema (`chpasswd`, `systemctl restart vsftpd`), e o isolamento multi-tenant usa interpolação de string do slug em SQL raw — vetor potencial de SQL injection. **Top-3 vereditos imediatos:** (1) instalar e habilitar o `cron` hoje; (2) colocar as portas 4000 e 8000 atrás do Traefik e removê-las do UFW; (3) aplicar `requireAdmin` em todos os endpoints que chamam `exec`/`spawn` no sistema. Sobre o proxy reverso — **Traefik v3.6.1 é o único componente público** (portas 80/443); Nginx roda apenas como sidecar dentro do container `netagent-frontend`, servindo estáticos na rede Docker interna.

## 3. Reverse proxy — resposta objetiva

**Traefik v3.6.1 é o proxy reverso público**, expondo 80/443 para a internet com TLS automático Let's Encrypt (ACME HTTP-01). Verificado em 2026-04-21 via:

- `docker ps`: único container bindando `0.0.0.0:80` e `0.0.0.0:443` → `traefik`.
- `curl -I https://agente.forumtelecom.com.br/` → HTTP/2 200, header `server: traefik`.
- `openssl s_client -connect agente.forumtelecom.com.br:443 -showcerts`: cadeia Let's Encrypt R12 válida, `notAfter=May 22 02:02:07 2026 GMT`.
- `docker logs traefik`: "Testing certificate renew..." diário, provider `letsencrypt.acme`.

**Nginx existe apenas como sidecar interno** rodando no container `netagent-frontend` (imagem `nginx:alpine`). Sua única função é servir os artefatos estáticos `frontend/dist` (SPA React) na rede Docker `netagent-net`, para que Traefik faça o reverse proxy. **Não há porta exposta ao host** pelo sidecar — o `docker ps` mostra apenas rede interna 172.18.0.x. Qualquer tráfego de usuário final passa obrigatoriamente pelo Traefik.

Detalhes em [REVERSE-PROXY.md](REVERSE-PROXY.md).

## 5. Detalhamento dos bugs

<a id="audit-bug-001"></a>
### AUDIT-BUG-001 — Cron daemon ausente
- **Severidade:** Crítico
- **Local:** servidor (Debian 12)
- **Evidência:** `systemctl is-active cron` → `inactive`; `systemctl status cron` retorna `Unit cron.service could not be found`; binário `crontab` ausente no PATH; diretório `/var/spool/cron/crontabs/` não existe.
- **Impacto:** Entries em `/etc/cron.d/docker-image-prune` e `/etc/cron.d/e2scrub_all` **NUNCA executam**. Disco do servidor tende a encher com imagens Docker antigas não removidas. Qualquer outro cron planejado em `server-setup.sh` ou `installer/scripts/install.sh` também está inoperante. Scheduler interno do Agent (APScheduler) cobre tarefas da aplicação — não é afetado.
- **Recomendação:** `apt install cron && systemctl enable --now cron`. Validar em seguida que `systemctl is-active cron` retorna `active` e que os arquivos em `/etc/cron.d/` executam (log `/var/log/syslog`).

<a id="audit-bug-002"></a>
### AUDIT-BUG-002 — API Node.js (porta 4000) aberta publicamente via UFW
- **Severidade:** Crítico
- **Local:** `ufw status` + `ss -tlnp`
- **Evidência:** UFW rules incluem `4000/tcp ALLOW Anywhere`; o processo Node bindando `*:4000` (0.0.0.0), não loopback.
- **Impacto:** Qualquer cliente na internet pode fazer requisições HTTP **direto na API Node, fazendo bypass do Traefik**, sem TLS, sem middlewares Traefik (rate limit, IP allowlist). JWT continua protegendo rotas autenticadas, mas a defesa em profundidade está ausente — endpoints públicos (como `/auth/login`) ficam diretamente expostos a brute force HTTP não-criptografado.
- **Recomendação:** `ufw delete allow 4000/tcp`; alterar bind do Express em `api/src/index.js` (ou equivalente) para `127.0.0.1`; expor por trás do Traefik com rota `/api/*` já existente.

<a id="audit-bug-003"></a>
### AUDIT-BUG-003 — Agent Python (porta 8000) aberta publicamente via UFW
- **Severidade:** Crítico
- **Local:** `ufw status` + `ss -tlnp`
- **Evidência:** UFW rules incluem `8000/tcp ALLOW Anywhere`; Agent FastAPI/uvicorn em `0.0.0.0:8000`.
- **Impacto:** Endpoints `/chat`, `/webhook`, `/skills/convert` acessíveis externamente sem TLS. `/chat` conecta com LLM — exploração externa pode **escalar custo OpenAI** arbitrariamente. Webhook Evolution recebe payloads do WhatsApp; se exposto, atacante pode injetar eventos forjados.
- **Recomendação:** `ufw delete allow 8000/tcp`; bind em `127.0.0.1:8000`; expor via Traefik em subpath/subdomínio com autenticação. Para comunicação API Node → Agent, chamar via rede Docker interna ou loopback.

<a id="audit-bug-004"></a>
### AUDIT-BUG-004 — 6 vulnerabilidades HIGH em dependências da API
- **Severidade:** Crítico
- **Local:** `api/package.json` + `npm audit`
- **Evidência:** `npm audit` reporta 9 vulnerabilidades (3 moderate, 6 high):
  - `path-to-regexp 8.0.0-8.3.0` — DoS via regex (GHSA-j3q9-mxjg-w52f, GHSA-27v5-c462-wpq7)
  - `socket.io-parser 4.0.0-4.2.5` — unbounded binary attachments (GHSA-677m-j7p3-52f9)
  - `tar <=7.5.10` — 6 CVEs encadeadas (path traversal, symlink poisoning, race condition em APFS)
  - `@mapbox/node-pre-gyp` → depende de `tar` vulnerável
  - `bcrypt 5.0.1-5.1.1` → depende do acima; upgrade para 6.0.0 é breaking change
  - `minimatch <=3.1.3` — 3 ReDoS (GHSA-3ppc-4f35-3m26, GHSA-7r86-cg39-jmmj, GHSA-23c5-xmqv-rm74)
- **Impacto:** `path-to-regexp` é parte do Express — rotas vulneráveis a DoS via payload engenheirado. `socket.io-parser` afeta WebSocket (chat, notificações). `tar` via bcrypt só corre em instalação nativa, menor risco em runtime, mas suja a árvore de dependências.
- **Recomendação:** `cd api && npm audit fix --force`. Como upgrade de bcrypt é breaking, rodar suíte de testes de autenticação e realizar rehash se necessário (bcrypt@6 mantém compat com hashes antigos, mas validar).

<a id="audit-bug-005"></a>
### AUDIT-BUG-005 — ACME token failure sugere renovação de cert em risco
- **Severidade:** Médio
- **Local:** `docker logs traefik`
- **Evidência:** `2026-03-25T05:05:01Z ERR Unable to get token error="missing token" providerName=acme`; certificados atuais expiram em **2026-05-22 (31 dias restantes)**. Linhas "Testing certificate renew..." diárias confirmam o scheduler ACME rodando, mas não há log de renovação bem-sucedida desde a emissão.
- **Impacto:** Se a próxima tentativa de renovação falhar pela mesma razão, os dois domínios ficarão sem HTTPS válido em ~30 dias → outage total dos painéis e webhooks.
- **Recomendação:** Verificar existência e permissões 600 em `traefik/certs/acme.json`; checar se a porta 80 está reachable externamente (desafio HTTP-01); forçar uma tentativa de renovação manual (restart Traefik ou excluir cert do `acme.json` para re-emissão) dentro dos próximos 10 dias (renewal começa aos 30 dias restantes); configurar alerta externo (UptimeRobot, Healthchecks.io) monitorando o `notAfter` dos certs.

<a id="audit-bug-006"></a>
### AUDIT-BUG-006 — Traefik reporta middleware `https-redirect@docker` não existe
- **Severidade:** Médio
- **Local:** `docker logs traefik` + `docker-compose.yml:56-59,137-139`
- **Evidência:** Log recorrente: `middleware "https-redirect@docker" does not exist entryPointName=web routerName=evolution-http@docker`. Apesar do erro, `curl -I http://agente.forumtelecom.com.br/` retorna 307 para HTTPS — redirect funciona em produção via outro mecanismo.
- **Impacto:** Inconsistência de configuração. Logs sujos dificultam debug futuro. Se o middleware que funciona hoje for removido, o redirect quebra sem aviso.
- **Recomendação:** Mover a definição do middleware `https-redirect` de labels Docker (local, escopada ao serviço que a define) para o file provider em `traefik/config/middlewares.yml` — assim fica disponível globalmente a todos os routers via `@file`.

<a id="audit-bug-007"></a>
### AUDIT-BUG-007 — Regra UFW para Redis (6379) é inconsistente com o bind real
- **Severidade:** Médio
- **Local:** `ufw status`
- **Evidência:** `6379/tcp ALLOW Anywhere` no UFW, mas Redis bind é `127.0.0.1:6379` (não escutando em interfaces externas).
- **Impacto:** Regra enganosa. Se algum operador alterar o bind para `0.0.0.0` sem revisar UFW, Redis passa a estar exposto com autenticação potencialmente fraca — e a regra UFW permite silenciosamente.
- **Recomendação:** `ufw delete allow 6379/tcp`. Redis deve ficar apenas na rede Docker interna.

<a id="audit-bug-008"></a>
### AUDIT-BUG-008 — MCPs (8001/8002) bindam em 0.0.0.0
- **Severidade:** Médio
- **Local:** `ss -tlnp` + `docker-compose.yml:154,171`
- **Evidência:** `0.0.0.0:8001` e `0.0.0.0:8002` no host. UFW default deny cobre hoje, mas a superfície no host está aberta.
- **Impacto:** Defesa em profundidade reduzida. Os MCPs executam comandos SSH em MikroTiks e servidores Linux dos clientes — exposição externa = execução de comandos arbitrários em infraestrutura de terceiros.
- **Recomendação:** No `docker-compose.yml`, alterar `ports:` de `"8001:8001"` para `"127.0.0.1:8001:8001"` (idem 8002). Como API/Agent chamam via `host-gateway`, ajustar para `172.17.0.1` ou colocar MCPs em rede Docker comum e usar DNS interno.

<a id="audit-bug-009"></a>
### AUDIT-BUG-009 — Traefik dashboard (8080) em 0.0.0.0
- **Severidade:** Médio
- **Local:** `ss -tlnp` + `docker-compose.yml:27`
- **Evidência:** `0.0.0.0:8080` listening. UFW não permite (default deny), mas o socket está aberto no host.
- **Impacto:** Se UFW falhar ou for desligado temporariamente (manutenção), dashboard Traefik fica acessível externamente. Mesmo com `api.insecure=false`, é uma superfície que não deveria existir.
- **Recomendação:** Em `docker-compose.yml`, trocar `"8080:8080"` por `"127.0.0.1:8080:8080"`. Acesso administrativo via SSH port-forward.

<a id="audit-bug-011"></a>
### AUDIT-BUG-011 — AxiosError timeout 120s na API
- **Severidade:** Médio
- **Local:** `pm2 logs netagent-api`
- **Evidência:** `AxiosError: timeout of 120000ms exceeded` em log.
- **Impacto:** Alguma chamada HTTP saindo da API (provavelmente para o Agent em 8000, um MCP, ou a OpenAI) excedeu 2 minutos. Request do usuário falha após espera longa — UX ruim, potencial thread-stall se síncrono.

<a id="audit-bug-012"></a>
### AUDIT-BUG-012 — Asyncio bug no Agent MonitorScheduler
- **Severidade:** Médio
- **Local:** `pm2 logs netagent-agent` + `agent/monitor.py`
- **Evidência:** `Exception ignored in: <coroutine object MonitorScheduler._poll_device at 0x...>` + `RuntimeError: no running event loop` + `ERROR:asyncio:Task was destroyed but it is pending!`
- **Impacto:** Coroutines não canceladas corretamente no shutdown. A longo prazo pode causar memory leak, ignorar falhas de polling de dispositivos, ou deixar sockets SSH pendurados.
- **Recomendação:** Em `agent/monitor.py` (classe `MonitorScheduler._poll_device`), usar `asyncio.create_task(...)` mantendo referência, e no método de shutdown aguardar `task.cancel()` + `await asyncio.gather(*tasks, return_exceptions=True)`.

<a id="audit-bug-019"></a>
### AUDIT-BUG-019 — `requireAdmin` middleware definido mas NÃO aplicado em endpoints sensíveis
- **Severidade:** Crítico
- **Local:** `api/src/middleware/` + rotas (identificado pelo subagente de API-ENDPOINTS.md)
- **Evidência:** Middleware `requireAdmin` existe no código mas não é aplicado em endpoints como `chpasswd` (troca de senha via POST) e `systemctl restart vsftpd`. Esses endpoints são protegidos apenas por JWT (`authMiddleware`), ou seja, **qualquer usuário autenticado de qualquer tenant** pode chamá-los.
- **Impacto:** **Privilege escalation**. Um usuário tenant normal pode trocar senha de contas do sistema e reiniciar `vsftpd`. Acesso a comandos de nível root via HTTP autenticado.
- **Recomendação:** Aplicar `requireAdmin` (ou `requireSuperAdmin` se existir) em **todos** os endpoints que chamam `exec`/`spawn`. Fazer varredura manual: grep `exec(` e `spawn(` em `api/src/` e listar as rotas que chegam nesses calls. Cada uma precisa de middleware admin.

- **Severidade:** Crítico
- **Local:** `api/src/middleware/tenantMiddleware` (identificado pelo subagente)
- **Evidência:** Em vez de `SET search_path` com quoting adequado ou Prisma namespaced client, o middleware interpola o slug do tenant diretamente em queries raw SQL subsequentes.
- **Impacto:** Se o slug do tenant puder ser controlado por input do usuário (mesmo indiretamente — ex.: rota de signup, rota de lookup por slug, importação de dados), existe vetor claro de **SQL injection**. Mesmo hoje, com slugs definidos no cadastro, qualquer fluxo que permita editá-los ou que passe slug em query string é potencial ataque.
- **Recomendação:** Validar slug com regex estrita `^[a-z0-9_]+$` no momento do cadastro e a cada uso; migrar para `SET search_path TO "{slug}",public;` usando quoting seguro de identificador (ou Prisma `$extends` com schema override). Testar explicitamente com payloads como `foo"; DROP TABLE users; --`.

<a id="audit-bug-021"></a>
### AUDIT-BUG-021 — `express-rate-limit` importado mas não aplicado
- **Severidade:** Médio
- **Local:** `api/src/` (identificado pelo subagente)
- **Evidência:** Pacote `express-rate-limit` está nas dependências e importado em algum módulo, mas não é usado como middleware em nenhuma rota.
- **Impacto:** **Nenhum rate limit na API** — brute force em `/auth/login`, abuse em `/chat` (custo OpenAI), DoS por volume são todos viáveis sem mitigação. Combinado com [AUDIT-BUG-002](#audit-bug-002) (porta 4000 pública), o risco é concreto.
- **Recomendação:** Aplicar `rateLimit({ windowMs: 60000, max: 5 })` em `/auth/*` e um limite proporcional em `/chat` (ex.: 30/min por user). Considerar `express-slow-down` para degradação gradual em vez de bloqueio hard.

<a id="audit-bug-022"></a>
### AUDIT-BUG-022 — Enumeração de usuário por timing em `/auth/login`
- **Severidade:** Médio
- **Local:** rota `/auth/login` (identificado pelo subagente)
- **Evidência:** Caminhos de código distintos quando o user existe vs. não existe — `bcrypt.compare` só é rodado se o user foi encontrado no DB. Atacante mede latência da resposta para enumerar usuários válidos.
- **Impacto:** Username enumeration → lista alvo para credential stuffing e brute force direcionado.
- **Recomendação:** Sempre executar `bcrypt.compare(senha, hashDummy)` quando user não existe (hash dummy com mesmo cost factor que os reais), ou garantir um tempo mínimo constante na rota (`await sleep(300)` se response ficar abaixo).

<a id="audit-bug-023"></a>
### AUDIT-BUG-023 — `wg_server.js` instancia PrismaClient próprio
- **Severidade:** Cosmético
- **Local:** `api/src/routes/wg_server.js` (identificado pelo subagente)
- **Evidência:** `new PrismaClient()` local em vez de reaproveitar o singleton global da aplicação.
- **Impacto:** Pool de conexões duplicado, potencial esgotamento em alta carga (cada PrismaClient abre seu próprio pool).
- **Recomendação:** Importar o singleton já usado no resto da API (`import { prisma } from '../db'` ou equivalente).

<a id="audit-bug-024"></a>
### AUDIT-BUG-024 — Drift fastapi entre `requirements.txt` (pin) e venv
- **Severidade:** Cosmético
- **Local:** `agent/requirements.txt` vs. `agent/venv/`
- **Evidência:** Pin `fastapi==0.115.4` no `requirements.txt`; venv atual tem `fastapi 0.129.0` instalado. Diversos outros pacotes outdated (pydantic, langchain, openai, etc. — ver seção 6).
- **Impacto:** Reprodução de ambiente divergente. Um `pip install -r requirements.txt` em instalação fresca vai puxar versão diferente da que roda em produção — comportamento pode ser diferente, inclusive em APIs breaking.
- **Recomendação:** Rodar `pip freeze > requirements.lock` e versionar o lock; ou atualizar o pin para a versão efetivamente em uso (`fastapi==0.129.0`). Longo prazo: adotar `pip-tools`/`uv`/`poetry`.

  - `agente.forumtelecom.com.br`: `notAfter = May 22 02:02:07 2026 GMT`
  - `agenteevo.forumtelecom.com.br`: `notAfter = May 22 04:09:15 2026 GMT`

### 7.2. Portas expostas externamente
Listening em `0.0.0.0` ou com regra UFW explícita:
- 22/tcp (SSH)
- 80/tcp, 443/tcp (Traefik)
- 2121/tcp (vsftpd)
- **4000/tcp (API Node — não deveria, ver [AUDIT-BUG-002](#audit-bug-002))**
- **8000/tcp (Agent Python — não deveria, ver [AUDIT-BUG-003](#audit-bug-003))**
- 40000-40500/tcp (FTP passive data range)
- 51820/udp (WireGuard concentrador)

Detalhes em [INFRASTRUCTURE.md](INFRASTRUCTURE.md).

### 7.3. Firewall UFW
- Status: **ativo** (default deny incoming).
- 1 regra inconsistente: `6379/tcp allow` (Redis bind é loopback) — [AUDIT-BUG-007](#audit-bug-007).
- 1 regra duplicada: `2121 (v6)` e `2121/tcp (v6)` — [AUDIT-BUG-018](#audit-bug-018).
- 2 regras que expõem serviços internos que deveriam estar atrás do Traefik: `4000/tcp` e `8000/tcp` — [AUDIT-BUG-002](#audit-bug-002), [AUDIT-BUG-003](#audit-bug-003).

### 7.4. Cron
**Daemon não instalado.** `systemctl is-active cron` → `inactive`, `Unit cron.service could not be found`. Scheduler interno do Agent (APScheduler) cobre tarefas de aplicação (monitoring, reminders) — não afetado. O que não roda: `docker-image-prune` (de `/etc/cron.d/docker-image-prune`) e `e2scrub_all` (de `/etc/cron.d/e2scrub_all`). Ver [AUDIT-BUG-001](#audit-bug-001).

### 7.5. FTP vsftpd
- UP na porta 2121/tcp.
- Passive data range 40000-40500/tcp aberto.
- Uso: recebe backups periódicos das configurações MikroTik dos clientes.
- `systemctl status vsftpd` → `active (running)` desde 2026-04-14.

### 7.6. Multi-tenant
- **2 schemas PostgreSQL:**
  - `public` — 6 tabelas de metadados globais (users, tenants, sessions, etc.)
  - `forumtelecom` — 15 tabelas do tenant atual (devices, backups, chat_history, knowledge_base, etc.)
- Isolamento feito via middleware Node que configura o schema ativo por request.
- **Risco:** interpolação de slug em query raw — ver [AUDIT-BUG-020](#audit-bug-020).

## 9. Limitações desta auditoria

O escopo foi **opção B** (leitura estática + execução read-only no servidor). **NÃO** foram realizados:

- Pentest ativo (injeção SQL real, XSS, auth bypass).
- Teste de isolamento multi-tenant real (tentativa concreta de vazar dados entre `forumtelecom` e `public` / outros schemas).
- Fuzzing de endpoints (payloads maliciosos, formatos inválidos, edge cases).
- Teste de carga / capacidade / stress.
- `pip-audit` no venv do Agent (ferramenta não instalada).
- Revisão aprofundada de permissões Linux, capabilities, seccomp/AppArmor dos containers.
- Verificação de backups efetivos (apenas validação da configuração do vsftpd).
- Revisão de secrets, rotação de chaves, exposição em env vars.
- Análise de logs por padrões de abuso / anomalia de acesso (além de scan superficial).
- Validação de compliance (LGPD, etc.).

Recomenda-se uma auditoria de segurança mais profunda (**opção C** do spec) com cadência anual após os itens P0 da seção 8 estarem resolvidos.
